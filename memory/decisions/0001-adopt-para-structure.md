# ADR-001: 采用PARA结构组织记忆库

**状态**: 已接受
**提出日期**: 2026-07-02
**最后更新**: 2026-07-02

---

### 背景

初始的记忆库结构是平铺的（code/errors/projects/tools），所有知识混合在patterns.json一个文件里。随着条目从5条增长到65条，出现了以下问题：
- 搜索成本增加：patents.json快到1000行了，每次查找得翻半天
- 领域混杂：Web渗透、游戏逆向、破解技术全在一个文件里
- 缺少决策记录：做了什么决策、为什么选这个方案，完全没有记录

### 方案对比

#### 方案A: PARA方法（选中）
- **描述**: Projects/Areas/Resources/Archives 四层 + 独立的Decisions目录
- **优点**: 顶级技术员验证过的成熟方法论，按行动性组织，不按文件类型
- **缺点**: 需要迁移现有数据

#### 方案B: 保持现状
- **描述**: 继续在patterns.json里加条目
- **优点**: 不用改
- **缺点**: 到100条就没法用了

#### 方案C: 全量RAG向量库
- **描述**: 把知识全塞进mcp-memory-graph的SQLite向量库
- **优点**: 搜索最强
- **缺点**: 过于依赖单个工具，GitHub备份不方便

### 决策

选方案A，PARA方法。原因是成熟、简单、和GitHub备份兼容。

### 后果

- **正面**: 结构清晰，条目按领域分散，好找
- **负面**: 需要迁移，旧的引用路径会变
- **风险**: 如果以后又长到1000条可能需要再拆分
- **缓解**: 每次新建条目时按领域分类放，不要全部塞一个文件
